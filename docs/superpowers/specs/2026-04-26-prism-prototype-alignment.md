# Prism 原型对齐设计文档

**日期**: 2026-04-26  
**项目**: Prism  
**目标**: 让 Tauri 应用完全对齐原型 `prism.html` 的所有功能和视觉

---

## 1. 背景与目标

### 1.1 当前状态
- Tauri 应用已实现核心编辑功能：编辑/分栏/预览、自动保存、KaTeX、Mermaid
- 产品方向已调整为 Typora 式单文档单窗口
- 但与原型 `prism.html` 相比，缺少桌面外壳、菜单系统、浮动工具栏、视觉精修

### 1.2 目标
让 Tauri 应用在功能、交互、视觉三个维度完全对齐原型 `prism.html`。

### 1.3 实施策略
采用**原型迁移式**：把原型中已验证的界面结构、交互层次、视觉语言系统性迁移到 Tauri 应用中，而不是参考原型重新设计。

---

## 2. 总体实施结构

为避免单个 spec/plan 失控，拆分为 4 个连续子项目，按顺序设计和落地：

### 子项目 1：桌面外壳对齐
**目标**: 让应用在第一眼观感上进入原型体系  
**内容**: titlebar、窗口控制按钮、menubar、菜单下拉框架、Fluent 风格窗口表面

### 子项目 2：编辑交互对齐
**目标**: 让应用"用起来像原型"  
**内容**: 浮动工具栏、快捷键、格式化动作、选区交互

### 子项目 3：视觉与布局对齐
**目标**: 让应用"看起来像原型"  
**内容**: mica 背景、layered surfaces、卡片容器、间距/圆角/阴影/滚动条精修

### 子项目 4：功能入口对齐
**目标**: 兑现原型菜单里的功能入口  
**内容**: 专注模式、全屏、偏好设置、打印/导出、搜索

**实施顺序**: 1 → 2 → 3 → 4

---

## 3. 子项目 1：桌面外壳对齐

### 3.1 目标边界
这一阶段只处理"应用壳"，目标是让 Tauri 应用的最外层结构对齐原型。

**要落地**:
- `window-shell`
- `titlebar`
- 窗口控制按钮
- `menubar`
- 一级菜单激活态
- 一级菜单下拉面板框架
- 浅色/深色下的 Fluent 外壳质感

**不要求**:
- 所有菜单项背后的功能都做完
- 只要求菜单能打开/关闭、层级正确、已实现功能能接进去、未实现功能先禁用或占位

### 3.2 组件拆分
拆成 4 个明确部件：

#### WindowShell
- 负责最外层圆角、边框、阴影、mica/layer 背景
- 包裹整个 App 主体
- 原型 `window-shell` 的正式迁移版本

#### TitleBar
- 显示 `Prism · 当前文件名`
- 左侧标题，右侧最小化/最大化/关闭按钮
- 适配 Tauri 窗口 API

#### MenuBar
- 横向渲染"文件 / 编辑 / 段落 / 格式 / 视图 / 主题 / 帮助"
- 管理当前激活的一级菜单
- 处理点击切换、外部点击关闭

#### MenuDropdown
- 渲染菜单项、分隔线、快捷键、子菜单箭头、disabled 状态
- 先支持一级菜单下拉
- 子菜单这阶段只保留视觉入口，不要求全部做成多层弹出

### 3.3 文件结构
新增跨域 shell 层：

```
src/components/shell/
├── WindowShell.tsx
├── TitleBar.tsx
├── MenuBar.tsx
├── MenuDropdown.tsx
├── menuData.ts
└── types.ts (必要时)
```

**为什么单独建 shell 层**:
- 这几部分不属于文档、工作区、编辑器任何一个单独领域
- 它们是整个桌面应用的外层壳
- 避免把 menubar 塞进 workspace 或 titlebar 塞进 editor

### 3.4 与现有结构的集成方式
当前应用主骨架保留：
- Sidebar
- DocumentView
- StatusBar

只是把它们从"直接挂在 App 根节点下"改成"挂在 WindowShell 里"。

结构变成：
```
WindowShell
├── TitleBar
├── MenuBar
├── workspace body
│   ├── Sidebar
│   └── DocumentView
└── StatusBar
```

**好处**:
1. 不动现有编辑域主链路
2. 不打散 sidebar / document / statusbar 的职责
3. 后面子项目 2、3、4 都可以继续在这个外层壳上迭代

### 3.5 状态与数据流
- `App.tsx` 负责把当前文档名、主题、已有动作传进 shell 组件
- `MenuBar` 内部只管理"当前展开哪个菜单"
- `menuData.ts` 只定义菜单结构，不直接调用业务逻辑
- 真正的动作绑定由 `App.tsx` 注入，菜单层不反向依赖 document/workspace/store

### 3.6 风险控制
最大风险：把原型视觉结构搬进来时破坏现有 `App.tsx` 的编辑器主布局。

**控制方式**:
- 采用"外层包裹替换"
- 保留现在的 `Sidebar + DocumentView + StatusBar`
- 只把它们放进新的 `WindowShell / TitleBar / MenuBar` 骨架里
- 先不重写编辑域内部结构

这样能保证：桌面外壳先对齐，但不会把当前已可用的编辑链路打散。

### 3.7 关键约束
这一阶段**不重构 App 的业务状态管理**。  
只做"外层壳替换"和"菜单骨架接入"。  
风险控制在 UI 骨架层，不碰坏现在已经可用的文件打开、编辑、预览、保存链路。

## 4. 子项目 2：编辑交互对齐

### 4.1 目标
让编辑体验对齐原型，重点是"用起来像原型"。

### 4.2 核心内容

#### 浮动工具栏
- 选中文本后出现
- 提供加粗、斜体、行内代码、链接、引用等快捷格式化
- 定位跟随选区
- 原型中的 `floating-toolbar` 迁移

#### 快捷键系统
- 把原型 `menuData` 里定义的快捷键真正接通
- Ctrl+B / Ctrl+I / Ctrl+S 等
- 需要在 Tauri 环境下正确拦截和响应

#### 格式化动作
- 把原型里的 `formatText` 逻辑迁移到 Tauri 应用
- 支持对 CodeMirror 编辑器的选区操作
- 保持编辑器焦点和光标位置

### 4.3 边界
- 不做菜单项背后的"新建文件""打开最近"这些功能入口
- 只做"编辑区内的交互增强"

---

## 5. 子项目 3：视觉与布局对齐

### 5.1 目标
让视觉质感对齐原型，重点是"看起来像原型"。

### 5.2 核心内容

#### Fluent Design 质感
- mica 背景
- layered surfaces（半透明分层）
- backdrop-filter 模糊和饱和度
- 精细的阴影层级

#### 编辑框与预览框一致性
- 确保编辑模式、分栏模式、预览模式下的卡片容器、留白、圆角、阴影完全一致
- 精修之前调整的"编辑框居中"

#### 细节收口
- 滚动条样式
- 间距系统
- 圆角系统
- 边框透明度
- hover / active 状态

### 5.3 边界
- 不新增功能
- 只做视觉还原和一致性

---

## 6. 子项目 4：功能入口对齐

### 6.1 目标
把原型菜单里的占位功能真正做出来。

### 6.2 核心内容

#### 专注模式
- 编辑区留白增大
- 隐藏 sidebar 和 statusbar
- F8 切换

#### 全屏模式
- 调用 Tauri 窗口 API
- F11 切换

#### 偏好设置
- 设置对话框
- 主题、字体、自动保存间隔等

#### 打印 / 导出增强
- 打印功能
- PDF 导出（如果 HTML 导出已有，这里补 PDF）

#### 搜索功能
- 工作区全局搜索
- 当前文档内搜索

### 6.3 边界
- 这一阶段才开始做"菜单项背后的功能"
- 前三个子项目只做外壳、交互、视觉

---

## 7. 错误处理

### 子项目 1
- 菜单下拉定位失败 → 回退到默认位置
- 窗口控制按钮调用 Tauri API 失败 → Toast 提示

### 子项目 2
- 浮动工具栏定位失败 → 不显示
- 格式化动作失败 → 保持原内容，Toast 提示

### 子项目 3
- 视觉降级：如果浏览器不支持 backdrop-filter，回退到纯色背景

### 子项目 4
- 功能调用失败 → 错误对话框
- 搜索无结果 → 空状态提示

## 8. 测试策略

### 子项目 1
- 手动测试：菜单打开/关闭、窗口按钮、主题切换
- 视觉回归：截图对比原型

### 子项目 2
- 手动测试：浮动工具栏出现/消失、快捷键响应、格式化效果
- 单元测试：格式化逻辑

### 子项目 3
- 视觉回归：截图对比原型
- 响应式测试：不同窗口尺寸

### 子项目 4
- 功能测试：专注模式、全屏、偏好设置、搜索
- 集成测试：菜单动作端到端

---

## 9. 实施顺序总结

1. **子项目 1**：先把外壳做对，让应用看起来像桌面应用
2. **子项目 2**：再把编辑交互做对，让应用用起来像原型
3. **子项目 3**：然后做视觉精修，让质感对齐
4. **子项目 4**：最后补功能入口，让菜单项可用

每个子项目独立设计 → 独立实施 → 独立验收。

---

## 10. 设计结论

采用原型迁移式，分 4 个子项目连续落地，先外壳、再交互、再视觉、最后功能入口。  
每个子项目边界清晰，风险可控，能够逐步让 Tauri 应用完全对齐原型。
